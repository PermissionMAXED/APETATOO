# APETATO

APETATO is an original action-roguelite arena-survivor game (monkeys, bananas, jungle weapons, chaotic waves) built with **Vite + three.js (r160) + vanilla JavaScript ES modules**. All content is procedural — there are **no binary assets** (no images/models/audio files); models are built from three.js primitives and all audio is synthesized via the WebAudio API.

The game lives in the **`APETATO/`** subdirectory (not the repo root).

## Cursor Cloud specific instructions

- **Where the app lives:** everything is under `APETATO/`. Run commands from there (e.g. `cd APETATO`). Dependencies (`three`, `vite`) are installed via `npm install` in `APETATO/` (handled by the startup update script).
- **Run the game (dev):** `cd APETATO && npm run dev` (Vite dev server, defaults to port 5173; use `-- --host 127.0.0.1 --port 5199` to pin it). Build: `npm run build`. Preview a build: `npm run preview`. There is **no lint step configured** (no ESLint in the repo).
- **Headless regression test:** `cd APETATO && node scripts/headless_run_test.mjs <seed> --quiet` runs a full simulated classic run to victory/defeat with no browser (great for verifying gameplay/balance logic quickly). The built-in auto-pilot is intentionally crude — it wins some seeds and dies on others (that reflects roguelite variance, not necessarily a bug). Use it to catch crashes/regressions and to sanity-check balance across several seeds.
- **IMPORTANT gotcha — software GL slow-motion:** this VM has no GPU, so Chrome renders the three.js canvas through software GL at roughly ~1 fps. The engine (`src/core/engine.js`) uses a fixed 1/60 timestep with a catch-up cap (max 5 steps/frame, then it drops time to avoid a spiral of death). At ~1 fps this makes the *simulation* advance in slow-motion in a real browser on this VM — the wave timer will appear nearly frozen and you may not reach the shop/boss in real time. This is an environment limitation, **not** a game bug. To verify full-run correctness/balance, prefer the headless test above; to demo the UI, note that the HTML/CSS menus/shop/HUD are fast even when the 3D canvas is slow. On real hardware the game targets 60 fps with hundreds of enemies.
- **Debug hook:** in the browser, `window.APETATO_DEBUG` exposes `{ bus, states, save, input, game }` for driving/inspecting the game from the devtools console.
- **Architecture (high level):** `src/core/*` = kernel (engine loop, event bus, state machine, seeded RNG, pooling, input, save, stat model). `src/game/*` = simulation systems (run loop, weapons, projectiles, combat, enemy AI, effects DSL, statuses, spawner/waves, bosses, shop logic, level-ups, synergies, companions/pets, hazards). `src/content/*` = pure data (22 characters, 90+ weapons, 150+ items, enemies/elites/bosses, 10 arenas, synergies, upgrades, modes) interpreted by the systems. `src/render/*` = three.js layer (InstancedMesh + object pooling, procedural models, particles, VFX, 2D-canvas damage numbers, arena builder) behind one `renderApi`. `src/ui/*` = HTML/CSS screens (menu, char select, mode/challenges select, HUD, level-up, shop, pause/settings, game-over/summary, leaderboard, stats, achievements, toasts). `src/audio/*` = procedural WebAudio SFX + music. `src/meta/*` = golden-banana progression, unlocks, achievements, lifetime stats, daily runs.
- **Data-driven content:** to add/tune weapons, items, characters, enemies, arenas, or modes, edit the frozen data objects under `src/content/`. Each content `index.js` runs a dev-only `validate()` (console warnings) that enforces unique ids and legal stat/behavior keys. Systems interpret the data, so new content generally "just works" if it conforms to the shapes.
- **Save data:** persisted in `localStorage` under `apetato_save_v1`. Wiping it triggers a clean first-run (3 default characters, Banana Grove arena, Classic mode).
- **Persistent state across VM snapshots:** none required beyond the repo + `node_modules`. No external services, databases, or secrets are needed.
